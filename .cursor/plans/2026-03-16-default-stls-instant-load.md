# Pre-rendered Default STLs for Instant Page Load

**Date**: 2026-03-16  
**Status**: completed

## Goal
Eliminate the render server call on initial page load by pre-rendering default-parameter STLs as static files served from S3/CloudFront. The finger model loads instantly instead of waiting 30-60s for the server.

## Steps
1. Create `scripts/generate_default_stls.py` — renders all 9 parts with default params using OpenSCAD
2. Add `generate-default-stls` Makefile target
3. Modify `app.js`: add `loadDefaultStls()` that loads static STLs from `/defaults/{part}.stl`; call it instead of `triggerRender()` when no hash/draft state exists
4. Generate the STLs locally (~38MB total, EXTRAMEDIUM quality matching preview server)
5. Deploy to S3 and verify via CloudFront

## Key decisions
- Chose `/defaults/` path instead of `/render/default/` because CloudFront proxies `/render/*` to EC2
- Used same render quality (EXTRAMEDIUM) as the live preview server for identical visual fidelity
- STLs are ASCII format (OpenSCAD default) — Three.js STL loader auto-detects and handles both formats
- Added `--exclude "defaults/*"` to the first `s3 sync --delete` command to prevent the static STLs from being deleted during the `web/` sync step

## Files affected
- `scripts/generate_default_stls.py` — new script to render default STLs
- `web/app.js` — added `loadDefaultStls()`, changed default load path to skip render
- `Makefile` — added `generate-default-stls` target, exclude `defaults/` from delete sync
- `web/static/defaults/*.stl` — 9 pre-rendered STL files (not committed, generated by script)

## Outcome
Works as expected. Page loads and the 3D finger appears instantly from CDN-served static STLs. No render server contact needed for default params. Users who change params still get the render overlay prompting them to re-render.

### Follow-up fixes (same session)
- **Bumper toggle missing**: CloudFront routes `/api/*` and `/params/*` to EC2, so static JSON in S3 was never served. Fixed by generating a `config-bootstrap.js` that embeds parts/params data as `window.__STATIC_PARTS__` / `window.__STATIC_PARAMS__`. The `api.js` checks these before making XHR calls.
- **STL parse errors**: ASCII STLs from OpenSCAD were misidentified as binary by Three.js STL loader (race condition with CloudFront caching). Switched to binary STL format (`--export-format binstl`), reducing total size from 38MB to 6.4MB.
- **Bumper position**: Shifted bumper X offset from 1.0 to 1.5 to prevent middle section clipping through it.
- **Part colors**: Replaced old dark brown / blue scheme with warm sandy/almond hard parts vs deep teal soft parts for clear visual distinction between rigid (PLA) and flexible (TPU) components.
